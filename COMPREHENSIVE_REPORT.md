## 🏗️ Architectural Transition

### 1. Cloudflare Edge Bridge (Middleware)
We have standardized on `middleware.ts` using the `edge` runtime. While Next.js 16 introduces `proxy.ts`, the `next-on-pages` bridge requires the standard convention to correctly synthesize the global Edge Worker.
- **Impact**: 100% CI/CD compatibility with Cloudflare Pages.
- **Config**: Root-level `middleware.ts` with `edge` runtime.

### 2. Edge-First Hydration
The Dashboard was refactored to handle URL-based state via a `<Suspense>` architecture.
- **Problem**: `useSearchParams()` was causing a bailout during static generation.
- **Solution**: Decoupled the hydration logic into a `DashboardContent` component with a high-fidelity glassmorphic skeleton fallback.

## 🛡️ Security & Integrity

### 1. Data Sanitization (DOMPurify)
All shared intelligence nodes are now passed through a server-side sanitization layer (`lib/sanitization.ts`) before being rendered.
- **Protection**: Mitigates XSS in shared public notes.
- **Optimization**: Sanitization occurs on the Edge during the SSR phase for zero-latency compute.

### 2. Edge Runtime Enforcement
Strict enforcement of the stable `edge` runtime across all dynamic routes ensures the application does not accidentally fall back to Node.js environments, maintaining 100% compatibility with Cloudflare Pages.
